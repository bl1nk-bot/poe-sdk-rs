# 🤖 คู่มือการรีวิวโค้ดสำหรับ AI Reviewer (AI Code Review Guidelines)

คู่มือนี้จัดทำขึ้นสำหรับบอท/เอเจนต์ AI ที่ทำหน้าที่รีวิว Pull Request (PR) และรหัสโค้ดใน Repository `bl1z` เพื่อให้บอทเข้าใจเกณฑ์การรีวิว รูปแบบข้อความ และวิธีการให้ความเห็น (Comment) อย่างเป็นระบบและตรงตามมาตรฐานของโครงการ

---

## 🎯 1. กฎเหล็กของ Repository (Hard Rules)

เอเจนต์ AI ต้องตรวจสอบจุดเหล่านี้อย่างเคร่งครัด หากพบการฝ่าฝืนให้ถือว่าเป็นข้อผิดพลาดระดับวิกฤต (Critical Blockers) ทันที:

1. **ห้ามใช้โค้ด `unsafe` โดยเด็ดขาด (Zero Unsafe Code):** 
   * ตรวจสอบว่าไม่มีคีย์เวิร์ด `unsafe` หรือบล็อกที่ไม่ปลอดภัยในโค้ด Rust
2. **การจัดการข้อผิดพลาดและข้อความแจ้งเตือนต้องเป็นภาษาไทย:**
   * ข้อความแสดงข้อผิดพลาดของระบบใน [src/error.rs](file:///D:/dev/10/poe-sdk-rs-fix-pr10/src/error.rs) หรือที่แสดงผลผ่าน `FormulaError` ทั้งหมด **ต้องเขียนเป็นภาษาไทย**
   * ต้องกำหนดรหัสข้อผิดพลาด (Error Code) ให้ตรงตามกลุ่มที่กำหนดไว้ในเอกสารสเปก:
     * `E1xx`: ข้อผิดพลาดขั้นตอนวิเคราะห์คำศัพท์ (Lex)
     * `E2xx`: ข้อผิดพลาดขั้นตอนวิเคราะห์ไวยากรณ์ (Parse)
     * `E3xx`: ข้อผิดพลาดขั้นตอนประเมินผลสูตร (Eval)
     * `E4xx`: ข้อผิดพลาดความเข้ากันได้ของชนิดข้อมูล (Type)
     * `E5xx`: ข้อผิดพลาดเกี่ยวกับการเรียกใช้ฟังก์ชัน (Function)
     * `E6xx`: ข้อผิดพลาดเกี่ยวกับบริบทตัวแปร (Context)
     * `E7xx`: ข้อผิดพลาดการทำ Serialization/Deserialization (Serialization)
     * `E8xx`: ข้อผิดพลาดเกี่ยวกับระบบปลั๊กอิน (Plugin)
3. **ความเข้ากันได้ของรุ่น Rust (MSRV):**
   * โค้ดทั้งหมดต้องเขียนด้วยไวยากรณ์ที่รองรับและคอมไพล์ผ่านบน **Rust 1.90.0** ขึ้นไป
4. **เอกสารกำกับและ Doc-tests สำหรับ Public APIs:**
   * ฟังก์ชัน โครงสร้างข้อมูล (struct) และ Enum ที่ประกาศแบบ `pub` ต้องมี Doc Comments (`///` หรือ `//!`) ที่มีโครงสร้าง `# Arguments`, `# Returns`, และ `# Examples` ที่รัน doc-test ผ่านเสมอ

## 📊 2. การประเมินระดับความรุนแรง (Severity Calibration)

เมื่อรีวิวโค้ด ให้บอทจัดกลุ่มประเด็นที่พบตามระดับความสำคัญดังนี้:

| ระดับความรุนแรง | คำอธิบาย | ตัวอย่างประเด็นที่พบ |
| :--- | :--- | :--- |
| **🚨 Critical (วิกฤต / Blocker)** | ข้อผิดพลาดทางลอจิกขั้นรุนแรง, ความปลอดภัยหน่วยความจำ หรือขัดต่อกฎเหล็ก | - มีการใช้งาน `unsafe`<br>- เกิด Stack Overflow หรือประเมินค่าวนลูปไม่สิ้นสุด (เช่น ปัญหา Eager evaluation ใน recursion)<br>- ข้อความ Error ไม่เป็นภาษาไทยหรือผิดรหัส<br>- มีการระบุ `#[allow(...)]` เพื่อหลบเลี่ยง Clippy warning โดยไม่มีคอมเมนต์ชี้แจงเหตุผล |
| **⚠️ Warning (เตือน)** | โค้ดส่งผลเสียต่อประสิทธิภาพ หรือขาดความสมบูรณ์ | - มีการเรียกใช้ dynamic heap allocation (`Vec`, `String`, `Box`) โดยไม่จำเป็นในจุดที่ต้องการประสิทธิภาพสูง<br>- ฟังก์ชันใหม่ไม่มีชุดการทดสอบ (Unit tests) ครอบคลุม<br>- เอกสาร API ตกหล่นหรือไม่สอดคล้องกับโค้ดจริง |
| **💡 Suggestion (คำแนะนำ)** | ข้อเสนอแนะเชิงโครงสร้าง เพื่อความสวยงามหรือการบำรุงรักษาในอนาคต | - คำแนะนำเพื่อ Refactor ให้โค้ดกระชับขึ้น<br>- การปรับปรุงชื่อตัวแปรให้อ่านเข้าใจง่ายขึ้น<br>- *หมายเหตุ:* ไม่ต้องเตือนเรื่อง formatting หากเครื่องมือ `cargo fmt` ตรวจสอบผ่านแล้ว |

## 💬 3. รูปแบบการเขียนคอมเมนต์ของบอท (Comment Formatting Guidelines)

ในการรายงานผลบน Pull Request หรือระบบ Issue ให้เอเจนต์รายงานผลเป็น **ภาษาไทย** และจัดโครงสร้าง Markdown ให้สวยงามและสแกนง่าย ดังนี้:

### โครงสร้างการคอมเมนต์ที่กำหนด:

```markdown
## 🤖 ผลการรีวิวโค้ดโดย AI (Code Review Report)

{บทสรุปภาพรวม 2-3 ประโยคเกี่ยวกับ PR นี้ เช่น ผ่านเกณฑ์พื้นฐาน หรือพบจุดที่ต้องแก้ไข}

### 🚨 ข้อผิดพลาดระดับรุนแรง (Critical Issues)
*ถ้าไม่มี ให้ระบุว่า "ไม่พบข้อผิดพลาดระดับรุนแรง"*
- [ชื่อไฟล์.rs:Lxx](ลิงก์ไปยังโค้ด): {รายละเอียดปัญหาและอธิบายว่าทำไมถึงขัดต่อกฎเหล็ก พร้อมเสนอแนวทางแก้ไข}
  ```rust
  // โค้ดที่ควรแก้ไขหรือโค้ดที่ถูกต้องที่แนะนำ
  ```

### ⚠️ ข้อควรระวังและคำเตือน (Warnings)
*ถ้าไม่มี ให้ข้ามส่วนนี้ไป*
- [ชื่อไฟล์.rs:Lxx](ลิงก์ไปยังโค้ด): {คำอธิบายจุดที่บกพร่อง เช่น ขาด Unit test หรือประสิทธิภาพต่ำ}

### 💡 คำแนะนำเพิ่มเติม (Suggestions)
<details><summary>คลิกเพื่อดูคำแนะนำเพิ่มเติมสำหรับการปรับปรุงโค้ด</summary>

- {คำแนะนำและเทคนิคการเขียนโค้ดเพิ่มเติม}
</details>

### ⚙️ ผลการตรวจสอบทางเทคนิค (Technical Verification)
- [ ] **การจัดรูปแบบโค้ด:** `cargo fmt` (ผ่าน/ไม่ผ่าน)
- [ ] **การตรวจสอบคุณภาพโค้ด:** `cargo clippy` (ผ่าน/ไม่ผ่าน)
- [ ] **การทดสอบความถูกต้อง:** `cargo test` (ผ่าน/ไม่ผ่าน)
- [ ] **เอกสารของโค้ด:** `cargo test --doc` (ผ่าน/ไม่ผ่าน)
```

## 🧪 4. ขั้นตอนการตรวจสอบความถูกต้องสำหรับบอท (Verification Steps)

ก่อนที่เอเจนต์จะคอมเมนต์สรุปผลการรีวิว จะต้องทำการทดสอบและรันชุดคำสั่งตรวจสอบดังนี้เสมอ:

1. **ตรวจสอบความสะอาดของสไตล์โค้ด:**
   ```bash
   cargo fmt -- --check
   ```
2. **รัน Clippy ตรวจสอบคำเตือนและข้อผิดพลาด (ต้องไม่มี warning เล็ดลอด):**
   ```bash
   cargo clippy --all-targets -- -D warnings
   ```
3. **ตรวจสอบ Unit & Integration Tests ทั้งหมด:**
   ```bash
   cargo test --verbose
   ```
4. **ตรวจสอบ Doc-tests ของเอกสารกำลับ:**
   ```bash
   cargo test --doc
   ```
5. **ตรวจสอบและอัปเดต snapshot test หากมีการแก้ไขโครงสร้าง Error:**
   ```bash
   INSTA_UPDATE=always cargo test --test error_snapshots
   ```
